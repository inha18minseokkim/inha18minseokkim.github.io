MSA TF에서 내년도에 Public 망 추진을 검토한다고 하여, 현재 행내에 있는 기술 스택으로 구성해 본 아키텍처 구상안이다.

## 데이터 흐름

| 단계 | 설명 |
|------|------|
| 1. 데이터 수집 | Public 망에서 자유롭게 외부 데이터 수집 |
| 2. 임시 저장 | 저장소 또는 Kafka에 데이터 적재 |
| 3. ETL 처리 | Kafka Connect를 활용하여 내부망으로 자동 반입 |
| 4. DB 저장 | 내부 데이터베이스에 최종 저장 |

디테일은 달라질 수 있지만, 큰 틀에서는 이 구조를 벗어나지 않을 것으로 예상한다.

---

## 우려 사항

**정보보호 승인 여부**가 관건이다.

만약 이 구조가 승인되지 않는다면, 대안은 다음과 같다:

| 대안 | 설명 |
|------|------|
| EAI 우회 | EAI를 거치지 않고 대외계 직접 연동 |
| EAI OOM 해결 | EAI의 메모리 문제 해결 (단, 가용성 이슈는 여전히 존재) |

> OOM 문제를 해결하더라도 가용성 이슈는 남아있다. 그래도 안 되는 것보다는 낫다.
