## 람다

메서드로 전달할 수 있는 익명 함수를 단순화한 것. 
익명성 - 이름 없음
함수 - 특정 클래스에 종속되지 않음(메서드 아님)
전달 - 람다 표현식을 파라미터로 전달, 변수 저장
간결 - 익명클래스 구현 불필요

```java
Comparator<Apple> byWeight = new Comparator<Apple>() {
	public int compare(Apple a1, Appl2 a2) {
		return a1.getWeight().compareTo(a2.getWeight());
	}
}
```

vs

```java
Comparator<Apple> byWeight = (Apple a1, Apple a2) -> a1.getWeight()
												.compareTo(a2.getWeight());

Comparator<Apple> c = (a1,a2) -> a1.getWeight().compareTo(a2.getWeight);
//파이썬에서나 봤던것 처럼 람다에서는 형식 추론이 가능해짐. 쓸지 말지는 개발환경에 따라 판단 필요할듯.
```

람다의 경우 람다가 정의된 지역 변수의 값은 못바꿈 
→ 참조 투명성, 멱등성 보장을 위함 
→ 인스턴스변수는 힙에 저장, 지역변수는 스택에 저장 → 만약 지역변수 접근 가능하다면 스레드에서 실행되는 람다가 변수를 할당한 사라지는 스레드의 변수에 접근하면 문제생김. 따라서 복사본으로 가져와야함.



## 함수형 인터페이스

오직 하나의 추상 메서드만 지정 가능

```java
public interface Predicate<T> {
	boolean test (T t);
}
```

인터페이스 내에 하나의 추상메서드만 있다면 전체 표현식을 함수형 인터페이스의 인터페이스로 취급해서 람다로 추정 가능

### 예시

Predicate<T> : T → Boolean
Consumer<T> : T → void
Function<T,R> : T → R
Supplier<T> : () → T
@FunctionalInterface 어노테이션 사용하면 추상메서드 한개로 제한 가능
